## Start Here: The Problem Dataclasses Solve

Before showing you `@dataclass`, let me show you the pain it eliminates.

Say you want a class to hold data about a student:

```python
class Student:
    def __init__(self, name, age, grade):
        self.name = name
        self.age = age
        self.grade = grade

    def __repr__(self):
        return f"Student(name={self.name!r}, age={self.age!r}, grade={self.grade!r})"

    def __eq__(self, other):
        if not isinstance(other, Student):
            return False
        return (self.name == other.name and
                self.age == other.age and
                self.grade == other.grade)

```

```python
s1 = Student("Arsh", 20, "A")
s2 = Student("Arsh", 20, "A")

print(s1)           # Student(name='Arsh', age=20, grade='A')
print(s1 == s2)     # True

```

This works. But look at what you wrote:

* `__init__` — just assigns each field to `self`. 3 fields = 3 lines of copy-paste.
* `__repr__` — manually formats every field. 3 fields = 3 mentions.
* `__eq__` — manually compares every field. 3 fields = 3 comparisons.

Add a fourth field → update three methods. Add a fifth → update three methods again. Every field touches three places. This is the boilerplate problem.

## What @dataclass Actually Is — The Feynman Version

`@dataclass` is a **code writer**. That's it.

You tell it what fields your class has. It reads your field declarations and **writes `__init__`, `__repr__`, and `__eq__` for you automatically at class creation time.**

It is not magic. It is not a new kind of class. It is a decorator that inspects your class, generates method code, and injects it into the class — before you ever create an instance.

```python
from dataclasses import dataclass

@dataclass
class Student:
    name: str
    age: int
    grade: str

```

That's the entire class. `@dataclass` reads those three field declarations and generates:

```python
# Auto-generated by @dataclass — you never write these:
def __init__(self, name: str, age: int, grade: str):
    self.name = name
    self.age = age
    self.grade = grade

def __repr__(self):
    return f"Student(name={self.name!r}, age={self.age!r}, grade={self.grade!r})"

def __eq__(self, other):
    if other.__class__ is self.__class__:
        return (self.name, self.age, self.grade) == (other.name, other.age, other.grade)
    return NotImplemented

```

```python
s1 = Student("Arsh", 20, "A")
s2 = Student("Arsh", 20, "A")

print(s1)           # Student(name='Arsh', age=20, grade='A') — __repr__ free
print(s1 == s2)     # True — __eq__ free
print(s1.name)      # Arsh — __init__ free

```

**The type hints (`name: str`) are not enforcement.** Python does not check types at runtime. They are annotations — information for the dataclass decorator to read, and for your IDE and type checkers to use. You can write `name: str` and pass an integer — no error.

## How @dataclass Reads Your Fields — The Mechanism

When Python loads your class definition, `@dataclass` does this:

```text
Step 1: Scan the class body for annotations (name: type)
Step 2: Build an ordered list of fields from those annotations
Step 3: Generate __init__ that accepts each field as a parameter
Step 4: Generate __repr__ that formats each field
Step 5: Generate __eq__ that compares each field as a tuple
Step 6: Inject all generated methods into the class
Step 7: Return the modified class

```

```python
# You can inspect what dataclass found:
from dataclasses import dataclass, fields

@dataclass
class Student:
    name: str
    age: int
    grade: str

print(fields(Student))
# (Field(name='name', type=str, ...),
#  Field(name='age',  type=int, ...),
#  Field(name='grade',type=str, ...))

```

`fields()` returns the ordered tuple of field objects the decorator found. This is the source of truth it uses to generate everything.

## Default Values

```python
@dataclass
class Student:
    name: str
    age: int
    grade: str = "A"      # default — optional at construction

```

```python
s1 = Student("Arsh", 20)         # grade defaults to "A"
s2 = Student("Riya", 19, "B")    # grade overridden

print(s1)   # Student(name='Arsh', age=20, grade='A')
print(s2)   # Student(name='Riya', age=19, grade='B')

```

**The rule:** fields with defaults must come after fields without defaults. Same as regular function parameters.

```python
@dataclass
class Bad:
    grade: str = "A"    # has default
    name: str           # no default — comes AFTER — TypeError

```

## Mutable Default Trap — `field(default_factory=)`

This is where most people get burned. You cannot use a mutable default directly:

```python
# ── BROKEN ──
@dataclass
class Student:
    name: str
    scores: list = []    # TypeError: mutable default not allowed

# ── WHY ──
# Same mutable default argument problem from functions.
# [] created once, shared across all instances.

```

```python
# ── CORRECT ──
from dataclasses import dataclass, field

@dataclass
class Student:
    name: str
    scores: list = field(default_factory=list)
    # default_factory=list means: call list() fresh for each instance

```

```python
s1 = Student("Arsh")
s2 = Student("Riya")

s1.scores.append(95)
print(s1.scores)   # [95]
print(s2.scores)   # []  ← independent, not shared

```

`default_factory` takes a **callable** — called with no arguments for each new instance:

```python
scores: list  = field(default_factory=list)          # calls list()
tags:   list  = field(default_factory=lambda: ["student"])  # calls lambda
config: dict  = field(default_factory=dict)          # calls dict()

```

## `field()` — Full Control Over Individual Fields

`field()` is how you customize what the decorator does with each field:

```python
from dataclasses import dataclass, field

@dataclass
class Student:
    name: str
    age: int
    scores: list = field(default_factory=list)
    _id: int = field(default=0, repr=False)       # hidden from __repr__
    password: str = field(default="", repr=False, compare=False)
    # compare=False → excluded from __eq__ comparison

```

```python
s1 = Student("Arsh", 20)
s2 = Student("Arsh", 20)
s2.password = "different_password"

print(s1)           # Student(name='Arsh', age=20, scores=[])
                    # _id and password not shown — repr=False
print(s1 == s2)     # True — password excluded from comparison

```

**`field()` key parameters:**

| Parameter | What it does |
| --- | --- |
| `default` | Static default value |
| `default_factory` | Callable for fresh default each time |
| `repr` | Include in `__repr__`? Default True |
| `compare` | Include in `__eq__`? Default True |
| `init` | Include in `__init__`? Default True |

## `__post_init__` — Running Code After Auto-Generated `__init__`

The auto-generated `__init__` just assigns fields. If you need validation or derived fields, use `__post_init__`:

```python
@dataclass
class Student:
    name: str
    age: int
    grade: str = "A"
    grade_points: float = field(init=False)   # init=False → not in __init__ signature
                                               # set by __post_init__ instead

    def __post_init__(self):
        # Runs automatically AFTER __init__ assigns all fields
        if self.age < 0:
            raise ValueError(f"Age cannot be negative: {self.age}")

        # Compute derived field:
        grade_map = {"A": 4.0, "B": 3.0, "C": 2.0, "D": 1.0}
        self.grade_points = grade_map.get(self.grade, 0.0)

```

```python
s = Student("Arsh", 20, "B")
print(s.grade_points)   # 3.0 — computed in __post_init__, not passed in

try:
    bad = Student("X", -5)
except ValueError as e:
    print(e)            # Age cannot be negative: -5

```

**Execution order:**

```text
Student("Arsh", 20, "B")
    ↓
auto __init__ runs:
    self.name = "Arsh"
    self.age = 20
    self.grade = "B"
    self.grade_points = MISSING (init=False)
    ↓
__post_init__ runs automatically:
    validates age
    computes self.grade_points = 3.0

```

## `frozen=True` — Immutable Dataclass

```python
@dataclass(frozen=True)
class Point:
    x: float
    y: float

```

```python
p = Point(1.0, 2.0)
print(p.x)      # 1.0
p.x = 99        # FrozenInstanceError — cannot assign to frozen instance

```

`frozen=True` also generates `__hash__` — making the instance usable as a dict key or in a set:

```python
@dataclass(frozen=True)
class Point:
    x: float
    y: float

p1 = Point(1.0, 2.0)
p2 = Point(1.0, 2.0)

locations = {p1, p2}   # works — frozen dataclass is hashable
print(len(locations))  # 1 — p1 == p2, same hash

d = {p1: "start"}      # usable as dict key

```

Without `frozen=True`: `@dataclass` generates `__eq__` which disables `__hash__` — same rule you learned in the dunder protocols section.

## `@dataclass` Parameters — Full Control

```python
@dataclass(
    init=True,       # generate __init__?         default True
    repr=True,       # generate __repr__?         default True
    eq=True,         # generate __eq__?           default True
    order=False,     # generate __lt__, __gt__?   default False
    frozen=False,    # make immutable?            default False
)
class Student:
    ...

```

`order=True` generates comparison methods based on field order — useful for sorting:

```python
@dataclass(order=True)
class Student:
    grade: str    # compared first
    name: str     # compared second if grades equal

students = [Student("B", "Riya"), Student("A", "Arsh"), Student("A", "Zara")]
print(sorted(students))
# [Student(grade='A', name='Arsh'), Student(grade='A', name='Zara'), Student(grade='B', name='Riya')]

```

## Complete Example — Everything Together

```python
from dataclasses import dataclass, field

@dataclass(order=True)
class Student:
    # Ordering is based on fields in declaration order
    # sort_index controls what gets compared for ordering
    sort_index: float = field(init=False, repr=False)

    name: str
    age: int
    grade: str = "A"
    scores: list = field(default_factory=list)
    _internal_id: int = field(default=0, repr=False, compare=False)

    def __post_init__(self):
        if self.age < 0:
            raise ValueError("Age cannot be negative")
        grade_map = {"A": 4.0, "B": 3.0, "C": 2.0, "D": 1.0}
        self.sort_index = grade_map.get(self.grade, 0.0)

    def add_score(self, score: float):
        self.scores.append(score)

    def average(self) -> float:
        return sum(self.scores) / len(self.scores) if self.scores else 0.0


s1 = Student("Arsh", 20, "A")
s2 = Student("Riya", 19, "B")
s3 = Student("Zara", 21, "A")

s1.add_score(95)
s1.add_score(88)

print(s1)                  # Student(name='Arsh', age=20, grade='A', scores=[95, 88])
print(s1.average())        # 91.5
print(s1 == s3)            # False — different names
print(sorted([s2, s1, s3]))# sorted by grade_points (sort_index)

```

## Mental Model Summary

```text
@dataclass is a code writer, not a new class type.

It reads your field annotations at class creation time
and generates __init__, __repr__, __eq__ for you.

field()          → customize individual field behavior
__post_init__    → validation and derived fields after __init__
default_factory  → fresh mutable default per instance (not shared)
frozen=True      → immutable + hashable
order=True       → sortable via __lt__, __gt__

Type hints       → not enforced at runtime
                   read by @dataclass to find fields
                   used by IDEs and type checkers

```

## Answer Key & Deep Dives

### Q1 — The Mutable Default Trap

**The Error:** `@dataclass` raises a `TypeError` at class definition time (before you even create an instance). It actively refuses to let you write `tags: list = []` because it knows it will trigger the shared memory trap.

**The Golden Rule of Mutable Defaults:**

> When an instance doesn't pass a value for a parameter that has a mutable default, that instance shares the SAME object as every other instance that also didn't pass a value. Mutating through one instance mutates the shared object — visible through all other instances that share it.
> **The Fix:** Never put mutable objects as defaults in the function signature. Create a fresh object inside the function body (or use `default_factory` in dataclasses) instead.

**Memory Map (Broken vs. Fixed):**

```python
# ── BROKEN: Shared Object in Definition ──
def __init__(self, tags=[]):      # [] lives here, created ONCE
    self.tags = tags              # All instances point to SAME []

# Memory representation:
#          default []  ←─────────────────┐
#               ↑                        │
# c1.tags ──────┘                        │
# c2.tags ───────────────────────────────┘


# ── FIXED: Fresh Object Per Instance ──
# using tags: list = field(default_factory=list)

# Memory representation:
# c1.tags ──► [list object A]   ← list() called fresh for c1
# c2.tags ──► [list object B]   ← list() called fresh for c2

```

### Q2 — `__post_init__` and Computed Fields

To compute a field based on other inputs, you must use `field(init=False)` to prevent `@dataclass` from adding it to the `__init__` parameters. Then, assign it inside `__post_init__`.

```python
from dataclasses import dataclass, field

@dataclass
class Person:
    first: str
    last: str
    full_name: str = field(init=False, repr=True)  # Excluded from __init__

    def __post_init__(self):                        
        # Runs AFTER first and last are assigned
        self.full_name = f"{self.first} {self.last}"

p = Person("Mohd", "Arsh")
print(p.full_name)   # Mohd Arsh — computed automatically

```

### Q3 — `frozen=True` and Hashability

**Why it fails:** By default, `@dataclass` generates an `__eq__` method. In Python, whenever you define a custom `__eq__` method, Python automatically sets `__hash__ = None`, making the object "unhashable". Dictionaries require their keys to be hashable.

**The Fix:** Use `@dataclass(frozen=True)`. This makes the instance immutable (you cannot change attributes after creation) and automatically generates both `__eq__` AND `__hash__`, allowing it to be used as a dictionary key or in a set.

```python
@dataclass(frozen=True)
class Point:
    x: float
    y: float

p = Point(1.0, 2.0)
d = {p: "location"}   # Works perfectly ✓

```

*(Note: If you absolutely need hashability WITHOUT immutability, you can use `@dataclass(eq=True, unsafe_hash=True)`, but this is generally discouraged as mutating an object after it is used as a dictionary key breaks the dictionary's internal logic.)*
